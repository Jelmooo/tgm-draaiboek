Van project aanmaken tot draaien op je server. Afgestemd op jouw eigen opzet.

ℹ️ Over de invul-plekken Op een paar plekken staat tekst tussen punthaken, zoals <projectnaam>, <repo-url> of <poort>. Vervang die (inclusief de punthaken) door je eigen waarde.

Kies bij de vragen: GitHub.com → HTTPS → Yes → Login with a web browser. Je krijgt een code die je in je browser plakt.

⚠️ Belangrijk als je met Google inlogt bij GitHub Dan heb je géén GitHub-wachtwoord. Git accepteert sowieso geen wachtwoorden meer. Authenticatie loopt daarom via gh auth login (hierboven) — niet via een wachtwoord.

1 Een nieuw project online zetten

Open een terminal in je projectmap (in Windows Verkenner: klik in de adresbalk, typ powershell, Enter) en typ:

git init -b main                  # maak hier een git-project
git add .                         # alle bestanden klaarzetten
git commit -m "Eerste versie"     # eerste opslagpunt
gh repo create <projectnaam> --public --source=. --push
Publiek of privé? --public = iedereen met de link ziet de code (simpelst: clonen op de server zonder inloggen). --private = alleen jij — maar dan moet je op de server inloggen met gh auth login of een token. Voor hobbyprojecten zonder geheimen is publiek prima.
🔑 Gouden regel Zet nooit wachtwoorden, API-sleutels of tokens in een repo (ook niet in een privé-repo). Houd die in een Docker-volume of in omgevingsvariabelen.

1 De dagelijkse routine

Telkens als je iets hebt veranderd en wilt vastleggen, typ je deze vier regels:

git status                      # wat heb ik veranderd?
git add .                       # alle wijzigingen klaarzetten
git commit -m "korte uitleg"    # opslagpunt maken (op je pc)
git push                        # naar GitHub sturen (nu online)

Voorbeeld:

git add .
git commit -m "Tekst op homepagina aangepast"
git push

2 Wat doet elk commando?

CommandoWat het doet
git status Laat zien wat er veranderd is. Verandert niets — gebruik het zo vaak je wilt.
git add . Neem alle wijzigingen mee in het volgende opslagpunt. (De punt = alles.)
git commit -m "…" Maak een opslagpunt met een kort briefje erbij. Blijft nog op je eigen pc.
git push Stuur je opslagpunten naar GitHub. Pas nu staat het online.
git pull Haal de nieuwste versie op vanaf GitHub (op een 2e pc of op de server).
git log --oneline Toon de lijst met al je opslagpunten (commits).
⚠️ De meest gemaakte beginnersfout commit en push zijn twee aparte stappen. commit = opslaan op je pc (werkt ook offline). push = naar GitHub sturen. Niets staat online totdat je git push hebt gedaan.
⭐ Onthoud de volgorde: eerst pushen, dan pullen De server haalt op wat er op GitHub staat. Vergeet je op je pc te pushen, dan trekt de server de oude versie binnen. Altijd eerst op je pc pushen, daarna pas op de server pullen.

CasaOSdraait als Docker

Je haalt de code op de server op en bouwt er een Docker-container van. Cloudflare zorgt voor de publieke URL. Dit gaat ervan uit dat je project een docker-compose.yml bevat met daarin de poort en het datavolume.

1 Eerste keer installeren

Ga naar je CasaOS-server (via SSH of de ingebouwde terminal-app) en typ:

# 1. Code ophalen in de map met live websites
cd "/Live Websites"
git clone <repo-url>
cd <projectnaam>

# 2. Docker-container bouwen en starten (sudo: jij bent gebruiker 'ubuntu')
sudo docker compose up -d --build

De eerste keer duurt het bouwen een minuutje (image downloaden + installeren). Aan het eind staat er zoiets als Container <naam> Started.

📌 Mappen met een spatie? Zet ze tussen quotes cd /Live Websites geeft een fout ("too many arguments"). Gebruik altijd cd "/Live Websites".
🔒 Is je repo privé? Dan vraagt git clone om in te loggen. Gebruik niet je GitHub-wachtwoord, maar log eenmalig in met gh auth login op de server — of zet de repo op publiek.

2 Controleren of het draait

sudo docker compose ps          # STATUS moet "Up" zijn
sudo docker compose logs --tail=40   # laatste meldingen / fouten
curl http://localhost:<poort>          # geeft de app antwoord?

Testen vanaf je gewone pc kan via http://<ip-van-de-server>:<poort>. De nette publieke URL regel je daarna in Cloudflare (laat die naar diezelfde poort wijzen).

💾 Hoe blijft mijn data bewaard?

In docker-compose.yml staat een volume, bijvoorbeeld:

volumes:
  - "/DATA/APPS DATABASE/<projectnaam>:/data"

Dit koppelt een map op de server aan de container. Daardoor:

  • Leeft je data op de server-schijf, búíten de container.
  • Blijft alles bewaard bij herstart, opnieuw bouwen, update en zelfs verwijderen van de container.
  • Maakt de app meestal alleen de eerste keer voorbeelddata aan; daarna wordt je eigen data nooit overschreven.
⚠️ Verander het volumepad niet zomaar Wijzig je het pad links van de :, dan kijkt de app ineens in een lege map en lijkt je data "weg". (Hij staat dan nog op het oude pad.)
📦 Back-up maken 
sudo cp "/DATA/APPS DATABASE/<projectnaam>/<databestand>" ~/backup-$(date +%F).json

⚠ Let op met de CasaOS-GUI

CasaOS toont je container als een app-tegeltje. Wat je daar wél en níét mee moet:

Wel via de GUINiet via de GUI
Starten / stoppen / herstarten, logs bekijken, app verwijderen. De instellingen (image, poort, volume, environment) bewerken.
Waarom niet? Een app die uit broncode wordt gebouwd, wordt beheerd door je docker-compose.yml. Dat bestand (in git) is de "waarheid". Wijzig je iets via de GUI en sla je op, dan ontstaat een conflict — bij de volgende docker compose up wordt het weer overschreven. Daarom toont het GUI-formulier ook een lege image: het kan zo'n app niet goed bewerken. Instellingen wijzig je dus in docker-compose.yml + git, niet in de GUI.

i Een update uitrollen in 2 stappen

Heb je iets aangepast en wil je dat op de server zien? Dat gaat altijd zo:

A
Op je pcpush naar GitHub
→
B
Op CasaOSpull + opnieuw bouwen
⭐ De volgorde is cruciaal Eerst stap A (pushen op je pc), pas daarna stap B (pullen op de server). Sla je stap A over, dan haalt de server de oude code op en lijkt je wijziging "niet door te komen".

A Op je pc — wijziging online zetten

git add .
git commit -m "wat je veranderd hebt"
git push

Check eventueel met git status: "up to date with origin/main" = veilig online.

B Op de server — ophalen + opnieuw bouwen

cd "/Live Websites/<projectnaam>"
git pull                        # nieuwste code ophalen
sudo docker compose up -d --build    # container opnieuw bouwen + starten
📦 Je gegevens blijven bewaard De data staat in een volume onder /DATA/APPS DATABASE/<projectnaam>, los van de code. Een update gooit je gegevens dus niet weg. (Zie tab ③ → "Hoe blijft mijn data bewaard?")
